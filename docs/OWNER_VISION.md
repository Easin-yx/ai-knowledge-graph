# 主人期望说明书（Owner Vision）

> **用途**：记录项目主人对知识图谱的视觉和结构期望。AI 在改任何图谱之前必须先读这份文档。
> **更新方式**：与主人逐图讨论后填写，每张图一次对话；讨论完一张，立刻补充对应章节。
> **优先级**：本文档的期望优先于通用规则（loop-engineering.md），但不违反 Harness 边界。

## Part 1：全局视觉期望（适用所有图谱）

> 以下为已确认的期望。每条后面标注「✅ 已实现」或「🔲 待实现」。

### 布局稳定性
- 🔲 每次刷新，节点位置保持一致，不随机跳动（空间记忆锚点）
  - 技术依据：P0 确定性径向布局已实现，待人工浏览器复核

### 主次分明
- 🔲 打开任何图谱，5 秒内能区分「主干」和「关联连线」
- ✅ 骨架边（实线/较粗/常显）与关联边（细线/静息态半隐）视觉已分层
- 🔲 默认状态下关联边不影响视线，hover 节点时才完整浮现

### 内容可读性
- 🔲 节点点开后，内容讲解按该节点的「角色」展示，不全部用同一套模板
  - 技术依据：OPTIMIZE_PLAN 的「节点按角色分形态」机制，编程图谱已试点，待推广

### 「打个比方」使用边界（全图通用）
- 🔲 类比（analogy）不是必填项：只在能显著降低理解门槛时使用，禁止为“有而有”的硬类比。
- 🔲 以下节点类型默认不写类比，优先写事实/参数/步骤：
  - 中后台工具节点（dashboard/cms/heatmap/pipeline 等）
  - 规则参数节点（冷却、消耗、倍率、判定窗口）
  - 清单/获取条件节点（掉落、解锁、章节条件）
- 🔲 以下节点类型可优先保留类比：
  - 抽象概念节点（concept/category）
  - 体验感知节点（叙事、沉浸、心流、能力表达）
  - 学习门槛较高但可用日常经验解释的机制节点
- 🔲 若某图谱类比过密，优先策略：
  1) 先按节点角色删除不适用类比；
  2) 再调整 `type -> cardArchetype` 映射，避免默认模板强行展示类比；
  3) 最后补充 facts/steps/contrast 等更可执行的信息块。

### 同级节点连线原则（通用边审查规则）

共享同一父节点的兄弟节点之间，**默认不需要额外连线**——父节点已经隐含了它们"同属一类"的关联。

同级边只在满足以下条件时才保留：

> **删掉这条边，会损失一个用父节点说不清的、有方向性的具体知识点。**

| 判断 | 处理方式 |
|---|---|
| 边的 label 泛化（"相关"/"对比"/"关联"，无实质内容）| 删掉，是噪音 |
| 边表达了有方向的因果/演化/替代/分叉知识点 | 保留，这条边本身就是知识 |

**反例（应删）**：`supervised_learning --对比-- unsupervised_learning`——父节点 `machine_learning` 已说明两者同属学习范式，对比关系是显然的，边无增量。

**正例（应留）**：`rnn --被取代于-- transformer`——父节点不会说明这个历史事件；`bert --对比-- gpt`——父节点不会说明编码器/解码器路线分叉。

### AI 图谱（`ai`）
**状态**：✅ 已整理（2026-06-27）

#### 结构感觉不对的地方

**① `transformer` 的入场路径太隐蔽**
- 现状：`transformer` 通过 `rnn → seq2seq → transformer` 这条历史演进链进入图谱，层级很深。
- 问题：Transformer 在今天的 AI 体系里是最核心的架构，但用户展开图时要经过多层点击才能看到它，入场层级和其重要性严重不匹配。
- 期望：`transformer` 应该有更显著的一级连接路径，不能只靠历史关联边进图。

**② `llm` 悬空，父节点层级太深**
- 现状：`llm` 有 14 条连线，是图里最重的 Hub 之一，但它通过 `llm__基于__transformer` 连入图——而 `transformer` 本身已经在图的很深处。
- 问题：用户展开图时，`llm` 出现的层级位置和它的重要性（AI 现阶段最核心的技术方向之一）完全不匹配。
- 期望：`llm` 应在更靠近根节点的层级出现。

**③ 训练工程实践与学习范式混在同一层**
- 现状：`machine_learning` 根节点直连了两类性质不同的节点：
  - 学习范式类：`supervised_learning`、`unsupervised_learning`、`reinforcement_learning`、`self_supervised_learning`、`deep_learning`
  - 训练工程类：`dataset_split`、`feature_engineering`、`hyperparameter_tuning`
- 问题：两类概念抽象层次不同，放在同一层视觉上层次不齐平。
- 期望：训练工程类节点应归入一个「训练实践」或「ML 工程」的中间 Hub，与学习范式分开一层。

**④ `diffusion_model` 归属混乱，同类生成式模型散落各处**
- 现状：`diffusion_model` 通过 `transformer__被借鉴于__diffusion_model` 关联边进图（技术借鉴关系）。但它的知识同类——`gan`（在 `deep_learning` 下）、`vae`（在 `neural_network` 下）——却在完全不同的分支里。
- 问题：同属"生成式模型"的三个节点（GAN/VAE/Diffusion）分散在三个不同的父节点下，没有"生成式模型"这个共同的概念归属层。
- 期望：考虑新增「生成式模型」Hub，统一归属 `gan`、`vae`、`diffusion_model`，再通过关联边表达各自的技术借鉴来源。

**⑤ `prompt_engineering` 的归属范围偏窄**
- 现状：`prompt_engineering` 通过 `ai_agent__依赖__prompt_engineering` 连入图，归属在 `ai_agent` 分支下。
- 问题：提示词工程不是 Agent 专属技能，所有使用 LLM 的场景都需要它；只挂在 `ai_agent` 下，概念范围感太窄。
- 期望：`prompt_engineering` 应有一条来自 `llm` 的直连骨架边，体现「LLM 的使用方式」这一更宏观的归属。

#### 期望补充的关系
- `llm` → `prompt_engineering`：LLM 的使用方式，骨架边
- 「生成式模型」Hub → `gan` / `vae` / `diffusion_model`：三者同属生成式范式，补一层归属

#### 不理解的连线
> 目前暂无，以上问题均为「有连线但归属/层级不对」或「缺少连线」。

#### 内容期望
> 待后续深入讨论补充。

### 黑神话悟空图谱（`black-myth`）
**状态**：📝 讨论中（2026-06-29）

#### 不理解的连线
- 目前主要问题不是「看不懂某条边」，而是历史版本中「技能信息集中在少数大节点」导致阅读跳跃。
- 本轮结论：优先通过新增技能 Hub + 子树重排解决层级问题，而不是继续堆关联边。

#### 期望补充的关系
- `combat_system` → `combat_skill_system`：战斗总框架与技能网络分层。
- `combat_skill_system` → 五大分支：棍法 / 法术 / 变化 / 法宝 / 召唤与精魄。
- `ux_ui_design`（升级为用户体验总入口）→ 叙事世界观 / 章节旅程 / 视听沉浸 / 能力表达。
- 新增体验节点与平台节点建立「支撑」映射（如本地化、资产管线、埋点、数值配置），保证双视角闭环。

#### 结构感觉不对的地方
- 旧版本战斗模块过于「机制粗粒度」：棍法仅靠 `gunshi_meter` / `resolute_strike` 两节点承载，细节不足。
- 旧版本 `ux` 偏 UI 表层，玩家向内容（叙事、旅程、视听、打法表达）密度不足。
- 旧版本技能分支未成网：棍法之外的法术/变化/法宝/召唤与精魄缺少统一归属层，易形成平铺与混层。

#### 内容期望
- 以玩家可理解为前提，补齐战斗技能体系，不只讲抽象机制名称。
- 棍法作为核心技能必须深拆（8-12 子节点），并与其他技能分支一并纳入统一结构。
- 来源必须可追溯：优先官网、BWiki，辅以 Game8、Maxroll；不以低可信聚合站作为事实依据。
- `build_expression` 定位为玩家体验结果层，不与 `combat_skill_system` 的机制层冲突。

#### 下一轮整改建议清单（按优先级）
- ✅ **P1-可读性**：已完成第一轮 `analogy` 清理（platform/参数型节点默认移除类比，减少硬比喻）。
- ✅ **P1-结构展示**：已复核 `skeleton-fragmented` 可视影响；当前接受「平台节点映射挂接」策略，不补噪音骨架边。
- ✅ **P1-角色模板**：已细化 `typeArchetypes` 与必要 `card` 覆盖，显著降低默认模板触发率。
- 🔲 **P1-真实性**：对技能节点执行“双来源优先”复核（官网/BWiki 为主，Game8/Maxroll 辅证），低置信条目留在 `low_confidence.md`。
- 🔲 **P2-深度扩展**：识破/棍势若继续下钻到帧级细节，必须先有高可信证据再入主图。
