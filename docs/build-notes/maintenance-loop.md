# Maintenance Loop · 图谱自维护循环规则

> 本文档 = **Agent 自驱动的图谱维护循环手册**。
> 目标不是「补全新大纲」，而是让 Agent **自行巡检已有图谱、发现问题、确认、修复、验证，再回到巡检**，直到上下文窗口将满才收尾。
> 总方法论见 [`loop-engineering.md`](loop-engineering.md)；各领域专属规则见 `<领域>/LOOP.md`。

---

## 0. 这个 Loop 解决什么问题

补全 loop（`<领域>/LOOP.md`）回答「图谱缺什么、要补什么」。
**维护 loop 回答「图谱已有的部分，哪里坏了、哪里不严谨、哪里讲不清」。**

它是一个**长跑型自巡检循环**：

> 巡检 → 发现问题 → 确认问题 → 确认修法 → 改 → 自验 → 回到巡检 …（循环）… → 上下文将满 → 收尾退出。

适用对象：**所有已通过 L0/L1 的图谱**（`ai` / `pm` / `black-myth` / `game-studio` / `programming-languages` / `grammar` / `expression`）。

---

## 1. 问题分类（巡检时找什么）

Agent 每轮巡检按下面三类找问题，**找到一类即可进入修复**，不要贪多。

| 类别 | 含义 | 典型信号 | 主要靠 |
|------|------|----------|--------|
| **C 代码 / 结构问题** | 数据文件或脚本层面的硬错误 | `validate` 报 error、`lint` 不过、悬挂边、重复 id、孤儿节点、type 非法、seed 非法 | `npm run validate` + `npm run lint` |
| **L 逻辑缜密性问题** | 内容自洽性、关系合理性 | 边语义矛盾（A 依赖 B 又 B 依赖 A 且无解释）、先修顺序成环、节点描述与其连边不一致、分级（萌新/进阶/精通）粒度错位、同一概念两处说法冲突 | 人读 + 验证子 Agent |
| **N 内容不全问题** | 该讲没讲清、覆盖到但深度不足 | 节点只有「是什么」无「谁用/上下游」、缺 `analogy` / `key_concepts` / `source`、`check-coverage` 有缺口、`low_confidence.md` 有待裁决项 | `check-coverage` + 评分卡 |

> **优先级**：C（硬错误）> L（逻辑）> N（内容）。
> 有 C 类未清时，**先清 C**——结构不对，谈内容没意义。

---

## 2. 单轮循环（Per-Iteration）

每一轮**只处理一个问题**（或一个紧密相关的问题簇），保证产出是**一个可 review 的小 diff**。

```
┌──────────────┐
│ 1. 巡检 / 选靶 │ ← 跑门禁 + 读 progress，锁定 1 个问题
└──────┬───────┘
       ▼
┌──────────────┐
│ 2. 确认问题   │ ← 复现/定位：它真的是问题吗？属 C/L/N 哪类？
└──────┬───────┘
       ▼
┌──────────────┐
│ 3. 确认修法   │ ← 定最小修改方案；拿不准 → low_confidence，跳过
└──────┬───────┘
       ▼
┌──────────────┐
│ 4. 执行修改   │ ← 改数据/脚本，遵守 §4 Harness
└──────┬───────┘
       ▼
┌──────────────┐
│ 5. 自我验证   │ ← validate + lint + (相关)check-coverage 全过
└──────┬───────┘
       ▼
┌──────────────┐
│ 6. 记账       │ ← progress.md 追加：发现了什么 / 怎么修的 / 验证结果
└──────┬───────┘
       ▼
   是否应继续？(§3) ──否──▶ 收尾退出
       │是
       ▼
   回到「1. 巡检 / 选靶」
```

### 2.1 每一步的硬要求

- **第 1 步 选靶**：每轮开始**必跑** `npm run validate` 和 `npm run lint`；若聚焦某图再跑 `npm run check-coverage:<mapId>`。从输出里挑**一个**最高优先级问题。
- **第 2 步 确认问题**：必须能说清「问题在哪个文件 / 哪个节点 id / 为什么是问题」。说不清 = 不算确认，重新巡检或换靶。
- **第 3 步 确认修法**：写出最小修改方案。**如果方案涉及编造事实、或拿不准是否真的是问题 → 不改，写进 `low_confidence.md`，本轮换下一个靶**。
- **第 4 步 改**：只动与本问题相关的最小范围。
- **第 5 步 自验**：门禁不全过，**不得进入下一轮**；修不好则回滚本轮改动，问题转入 `low_confidence.md`。
- **第 6 步 记账**：见 §5 模板。

## 3. 终止条件（何时退出循环）

满足**任意一条**即收尾退出，**不要再开新一轮**：

1. **上下文窗口将满**（主终止条件）——这是用户指定的边界。自评剩余上下文不足以再完整跑一轮「巡检→改→自验→记账」时，立即进入 §3.1 收尾。
2. **门禁全绿且本轮巡检零新问题**——`validate` / `lint` 0 error，聚焦图 `check-coverage` 0 缺口，且本轮未发现 C/L/N 任一问题。
3. **连续修复失败**——同一问题连续 2 轮自验不过且无法回滚到干净状态，停下交人工。
4. **用户喊停**。

### 3.1 收尾动作（退出前必做）

```
1. 确保工作区处于「门禁全绿」或「已回滚到全绿」状态——不留半截改动。
2. progress.md 写「本次维护会话小结」：本会话改了哪些轮、当前指标、未决问题。
3. low_confidence.md 汇总本次新增的待人工裁决项。
4. 给作者一个可 review 的 git diff（不自动 push）。
5. 写「下一步」：下个会话该从哪个问题/哪张图续跑。
```

> **绝不**：在上下文将满时硬塞一轮改一半的修改后退出。**宁可少修一轮，也要留干净状态。**

---

## 4. Harness · 维护循环边界

违反任意一条，**本轮作废、回滚**：

1. **禁止编造**——为「补全内容」而臆造技术细节/机制/产品名；核验不过 → `low_confidence.md`。
2. **禁止删节点/删边凑指标**——不靠删数据让 `validate` 变绿；error 要真正修对。
3. **禁止破坏式重写**——已准确的节点不为风格而全文重写；维护是**外科手术**不是推倒重来。
4. **禁止硬凑关系**——不为「让逻辑看起来自洽」而加不真实的边。
5. **一轮一问题一 diff**——不把多个不相关修复混进一轮。
6. **不动视觉契约**——`typeStyles`、主题色、PRD 范围除非用户明确要求。
7. **拿不准就降级**——任何不确定，进 `low_confidence.md`，绝不带着不确定信息污染图谱。
8. **保持模型与模式**——验证建议可换模型（减少同源偏见），但巡检/修复在同一上下文内连续进行。

## 5. 记账模板（progress.md 追加）

每轮结束在对应图谱的 `docs/build-notes/<领域>/progress.md` 追加一条：

```
### [维护] 轮 N · <yyyy-mm-dd>
- 类别：C / L / N
- 问题：<文件 / 节点 id / 一句话描述>
- 确认依据：<validate 输出 / 哪条评分卡维度 / coverage 缺口>
- 修法：<做了什么最小改动>
- 自验：validate ✅  lint ✅  check-coverage:<id> ✅(或 N/A)
- 残留：<无 / 进了 low_confidence 的项>
```

会话收尾再追加一条「本次维护会话小结 + 下一步」。

---

## 6. 启动语模板（新会话续跑）

```
读 docs/build-notes/maintenance-loop.md，对 <领域/all> 图谱执行维护 loop：
每轮先跑 validate + lint (+ check-coverage:<id>)，巡检 C/L/N 三类问题，
锁定一个问题 → 确认 → 定最小修法 → 改 → 自验门禁全绿 → 记 progress，
然后回到巡检继续循环。拿不准的进 low_confidence.md。
直到上下文将满或门禁全绿零新问题，按收尾流程退出并给一个可 review 的 diff。
```

## 7. 与补全 Loop 的关系

| | 补全 Loop（`<领域>/LOOP.md`） | 维护 Loop（本文档） |
|---|---|---|
| 触发 | 大纲有缺口 / 新图 | 图已成型，日常巡检 |
| 每轮目标 | 补一个主题/模块 | 修一个 C/L/N 问题 |
| 终止 | 覆盖率/评分卡达标 | 上下文将满 或 零新问题 |
| 产出 | 新节点/新边 | 小范围修复 diff |

两者共用同一套门禁（L0/L1/L2/L3）、同一套 Harness、同一套 `progress.md` / `low_confidence.md`。维护 loop 是图谱进入稳定期后的**常态化质量保障**。

---

## 8. 一句话总结

> **维护 Loop = 自驱动巡检（C/L/N）+ 单轮单问题 + 改前确认改后自验 + 拿不准就降级 + 每轮一个 diff + 上下文将满时干净收尾。**
