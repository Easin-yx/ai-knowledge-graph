# 知识图谱「养护」与重构计划（PLAN）

> 目标：把项目里的 7 张知识图谱，从「每次刷新乱跳、二级节点视觉杂乱」改造成
> **可记忆、可生长、结构自解释**的学习型图谱。本计划覆盖全部图谱，并给出
> 可落地的技术方案、数据规范、分图审计与实施路线。

## 1. 问题诊断（两个**独立**问题，勿混为一谈）

### 问题 A：每次刷新节点随机重排 —— 破坏「空间记忆」
- **现象**：除根节点外，节点位置每次刷新都不同。
- **根因**：`GraphCanvas.tsx` 用 **force simulation（力导向）** 定位，天生非确定性：
  - d3-force 内部 jiggle 使用 `Math.random()`；
  - `neighborAnchor()` 落点叠加了 `(Math.random() - 0.5) * 30` 抖动；
  - `nodeCacheRef` 仅存在于内存，刷新即清空。
- **为什么这是学习型图谱的头号 bug**：知识图谱学习法的核心红利是
  **空间记忆 / 记忆宫殿**——「KANO 在右上角那团」「需求链在左边那条线」。
  位置每次变，这条红利直接归零。

### 问题 B：二级节点之间互相连线，观感杂乱
- **认知澄清**：**「相连」恰恰是图谱相对于「思维导图（树）」的全部价值，不是缺陷。**
  - 思维导图是树：每个节点只有一个父亲，禁止横向连线。
  - 知识图谱是图：节点可以**既是某 hub 的孩子，又和兄弟节点有关系**。
  - 横向边表达的是「交叉关系」（如 `产品分析 —支撑→ 产品战略`），删掉等于把图退化成树，丢掉最值钱的部分。
- **真正的乱源**（见 §3/§4/§5）有三个，都**不是**「该合并」：
  1. **一级主干过多**（超出 7±2，见 §5 审计表）；
  2. **骨架边和关联边长得一样**，看不出主次（见 §4）；
  3. **布局非确定性**（即问题 A）放大了「乱」的主观感受。

## 2. 设计原则：养图谱六法

后续所有改造都服务于这六条心法，作为评审与扩图时的统一准绳：

1. **空间要稳**：确定性布局，位置=记忆锚点（→ §3）。
2. **中心克制**：一个根，5–7 个主干；超了就再分一层（→ §5）。
3. **粒度齐平**：同级节点的抽象层次要可比，避免「战略」和「某个按钮」并列。
4. **合并看语义，不看连线**：满足「说不清区别 / 完全被包含 / 从不单独出现」才合并（→ §6）。
5. **边分两类**：骨架边（实，管层级）vs 关联边（虚/hover 才显，管交叉）（→ §4）。
6. **渐进生长**：不一次全摊开，用现有展开交互一层层「长」出来（已具备，保留）。

## 3. 改造一：确定性径向布局（治「刷新乱跳」）

### 3.1 目标
**同一张图，每次、每台设备、每个人看到的版式完全一致。** 节点位置成为稳定的记忆锚点。

### 3.2 方案（推荐：确定性径向树 + 力导向仅做微调）
保留 `react-force-graph` 渲染与交互，但**位置由确定性算法预先计算并 `fx/fy` 钉死**，
力导向只用于展开瞬间的局部缓动，不再主导最终版式。

算法（在 `GraphCanvas` 内新增 `computeRadialLayout()`）：
1. 以 `seedIds` 为根做 **BFS**，得到每个节点的 `depth`（已有 `depthById`，复用）。
2. 对每一层，按**稳定排序键**给节点定角度：排序键 = `父节点角度` →
   `节点 id 字典序`（保证确定性，与数据书写顺序无关，增删节点时邻居不漂移）。
3. 半径 `r = depth * RING_STEP`；角度在父节点扇区内**均分**（树状放射）。
4. 把算得的 `(x, y)` 写入 `node.fx / node.fy`（钉死），力导向 `charge/link/collide`
   强度大幅调小，仅用于展开新节点时的入场缓动；`radial` 力可移除或保留极弱。

### 3.3 与渐进展开兼容（关键）
- 布局基于**全图**的 BFS 深度与排序预计算，但只渲染 `visibleData` 中的节点。
- 节点被展开时，落到它**预定的槽位**（而非随机邻居旁），实现「位置稳定 + 渐进生长」。
- 多连通分量：每个 seed 一个独立扇区/同心环组，互不重叠。

### 3.4 退化兜底
- 单层节点过多导致角度过密时，按 `RING_STEP` 自适应增大半径或允许同层多环。
- 提供 `layoutMode: 'radial' | 'force'` 开关，保留旧力导向作为回退（默认 `radial`）。

### 3.5 文件改动
- `src/components/GraphCanvas.tsx`：新增确定性布局函数；用 `fx/fy` 钉位；
  弱化/移除 `charge/radial` 主导；`onEngineStop` 的 `fitView` 逻辑保留。
- 不改数据，不改 `App.tsx` 的可见性逻辑。

### 3.6 验收
- 连续刷新 5 次，任意节点屏幕坐标一致（截图比对）。
- 展开/收回某节点，其余节点不发生整体重排。

### 3.7 备选档（如不想动布局算法）
- **持久化档**：把稳定后的 `x/y` 存 `localStorage`，刷新读回钉位。改动最小，
  但「首次」仍随机、跨设备不一致。仅作为临时缓解，不推荐作为终态。

## 4. 改造二：骨架边 vs 关联边 视觉分层

### 4.1 规则：按边的 `label` 分两类

| 类别 | 语义 | 典型 label | 渲染 |
| --- | --- | --- | --- |
| 骨架边（hierarchy） | 决定层级/归属 | `包含`/`分为`/`细分为`/`构成`/`组成`/`属于` | 实线、较粗、常显 |
| 关联边（association） | 交叉关系/补充 | `支撑`/`关联`/`对比`/`区别于`/`依赖`/`使用`/`基于`/`影响`/`退化`/`服务`/`借鉴`/`应用于`/`实现` | 细线/虚线、默认半隐，hover 或选中才高亮显形 |

### 4.2 实现
- 在 `KnowledgeEdge` 增加可选字段 `kind?: 'hierarchy' | 'association'`（默认按 label 推断）。
- `GraphCanvas` 的 `linkWidth` / `linkColor` / 默认透明度按 `kind` 区分；
  关联边静息态降透明度（如 0.25），`isLinkActive` 时恢复。
- 布局（§3）的「树骨架」**只用骨架边**做 BFS 定位，关联边不参与定位、只参与连线绘制，
  从根本上避免横向关联边把版式拉乱。

### 4.3 验收
- 静息态一眼能看出「主干树」；横向关联线退为背景，hover 节点时才浮现其关联。

- 检查每个 hub 的孩子是否抽象层次可比；不可比的下沉一层或上提。

## 6. 合并判据（什么时候才真的合并两个节点）

满足**任意一条**才合并，否则保持独立 + 用关联边连接：

1. 你说不清两者**一句话级别的区别**；
2. 一个**完全被另一个包含**（是「子集」而非「相关」）；
3. 两者**永远同时出现、从不需要单独区分**。

**反例（不该合并）：产品分析 vs 产品战略**
- `产品分析`(process)：给产品做体检，看数据、诊断现状 → 回答「现在怎么样？」
- `产品战略`(mindset)：为谁、解决什么、凭什么赢 → 回答「往哪走？」
- 二者已有 `产品分析 —支撑→ 产品战略` 的有向边，**这条边本身就是知识点**（分析为战略提供依据）。
- 结论：**保持独立，保留这条关联边。** 合并会抹掉这条因果关系。

> 各图 review 时，凡发现「想合并」的冲动，先套这三条判据；多数情况答案是
> 「不合并，改成一条有 label 的关联边」。

## 7. 工具与校验（保障可持续「养护」）

- 新增/扩展校验脚本（参考现有 `scripts/validate-graph.ts`）：
  1. 列出所有**度数 > 9** 的节点（中心收敛预警）；
  2. 检测**孤立节点 / 自环 / 重复边**；
  3. 校验边 `label` 是否落在「骨架/关联」词表内，未知 label 报警提示归类；
  4. 校验仅用骨架边时图是否连通（确保 §3 的树骨架能覆盖全部节点）。
- 词表维护在一个常量文件里，供校验脚本与 `GraphCanvas` 的 `kind` 推断共用。

## 8. 分阶段实施路线

| 阶段 | 内容 | 影响面 | 风险 |
| --- | --- | --- | --- |
| P0 | 确定性径向布局（§3） | 仅 `GraphCanvas.tsx` | 中（布局算法） |
| P1 | 骨架/关联边视觉分层 + `kind` 推断（§4） | `GraphCanvas` + `types` | 低 |
| P2 | 校验脚本：度数/连通/词表（§7） | `scripts/` | 低 |
| P3 | 按审计表逐图收敛中心扇出（§5），从 grammar / game-studio / pm 起 | 各 `maps/*.ts` 数据 | 低（纯数据） |
| P4 | 粒度对齐 + 合并判据复核（§5.3/§6） | 各 `maps/*.ts` 数据 | 低 |

> P0–P2 是「机制」，一次到位、全图受益；P3–P4 是「数据养护」，可分图渐进推进。

### 8.1 交叉引用：节点卡片「按角色分形态」（全局机制改造项）

> 起于编程语言图谱的优化，但本质是**跨全部 7 张图**的共性问题——详情卡片只有一套
> `摘要 + 类比? + 笔记? + 关键概念? + 翻面?` 骨架，强套所有认知角色的节点。

- **方案归档**：详见 [`../programming-languages/OPTIMIZE_PLAN.md`](../programming-languages/OPTIMIZE_PLAN.md) §9
  「全局通用化：节点卡片『按角色分形态』」。
- **机制概要**：抽象 `category / concept / mechanism / entity / practice` 五种卡片原型（archetype），
  渲染层做成「原型 → 显示哪些区块」的注册表；节点新增可选 `card` 标记 + `code / facts / steps / contrast`
  等结构化字段（全部可选、向后兼容，缺省由各图 `type → 原型` 映射推断）。
- **定位**：与本计划「养图谱六法」一脉相承，属**全局机制改造项**（对标 P0–P2 的「一次到位、全图受益」），
  编程图作为试点（pilot），其余 6 图按 `type → 原型` 映射渐进迁移、一图一提交。

## 9. 风险与回滚
- 布局算法改动风险最高：保留 `layoutMode` 开关，可一键回退旧力导向。
- 数据收敛只动 `edges` 的归属与少量 `kind` 字段，不删节点；每图改动独立可回退。
- 全程不改 `App.tsx` 的可见性/展开逻辑，渐进披露行为不变。

## 10. 总验收清单
- [ ] 任一图连续刷新，节点坐标稳定一致。
- [ ] 静息态主干树清晰，关联边退为背景、hover 浮现。
- [ ] 7 张图中心扇出均 ≤ 9（理想 5–7）。
- [ ] 校验脚本通过：无高度数预警、无孤立/重复、边 label 全部归类。
- [ ] 抽查若干「疑似可合并」节点，均按 §6 判据得到明确结论。
